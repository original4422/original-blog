# 博客建设复盘与教程双文章设计

## 目标

围绕本次个人博客从两个独立仓库演进为单一 monorepo、两套独立应用与 Pages 温备站的真实过程，发布两篇彼此互补的中文文章。第一篇解释决策与经验，第二篇提供可复现的工程方法。两篇文章同时构成 Task 9.4 所要求的真实公开内容变更，但旧 Pages 仓库只有在双端发布验证和 24 小时观察窗结束后才归档。

## 文章一：工程复盘

暂定标题为《把两个博客变成一个系统：我的个人网站重构复盘》。文章采用第一人称，以“个人博客首先是长期维护系统，而不是两个页面副本”为核心判断。结构从最初的 Vercel 与 Pages 双仓库开始，解释为什么平台冗余不等于内容可恢复，随后展开单一权威内容源、两套独立应用、独立自动部署、十五分钟新鲜度目标和人工恢复的取舍。

文章不写成成功学。Node 版本分叉、Pages `basePath`、移动导航、仓库重命名、Vercel Preview、`tsconfig.tsbuildinfo` 缓存等问题都会作为真实工程证据出现。结尾明确区分已经完成的能力与尚未完成的域名切换、异地 Git 镜像，避免把温备站夸大成完整灾备站。

## 文章二：可复现教程

暂定标题为《用一个 Monorepo 同时发布 Vercel 与 GitHub Pages》。文章面向熟悉 Git 与 Next.js、但没有做过双平台发布的个人开发者。它从目录结构开始，依次讲解 `packages/content` 的内容契约、`apps/vercel` 与 `apps/pages` 的平台边界、Pages 静态导出与项目路径、两套 `/version.json`、三条 GitHub Actions、Node 24 门禁、Playwright 移动导航测试和人工恢复验证。

教程不会复制仓库全部代码，而是给出足以理解和复现的最小配置、命令和失败模式。读者最终应能回答：哪些内容必须共享，哪些实现应保持独立，如何证明两个线上版本来自同一提交，以及为什么部署短暂不同不等于数据丢失。

## 图片与内容流

两篇文章共享同一批真实证据，但图片承担不同叙事任务：

- monorepo 架构图解释单一内容源如何驱动两套应用；
- Vercel 桌面端截图展示主站实际体验；
- Pages 移动端菜单截图展示温备站不是不可操作的静态副本；
- 发布收敛图解释同一 `main` 提交如何触发两条独立部署链并通过 `/version.json` 验证。

截图从当前公开网站采集，图表使用仓库内 SVG，全部存入 `packages/content/media/blog`。文章只引用规范 `/content/blog/...` 路径；Pages 的 MDX 图片组件负责在渲染边界增加 `/original-blog`，避免把平台路径写进内容。

## 验证与发布边界

内容层先运行 schema、slug、日期、媒体和单一来源审计。应用层在 nvm Node 24 下运行格式检查、类型检查、双生产构建、Pages 静态审计、版本审计与 Playwright。发布后分别检查两篇文章在 Vercel 与 Pages 的页面、图片、目录、Feed、Sitemap、搜索索引和 `/version.json`。

即时验证通过后开始 24 小时观察窗。窗口内旧 Pages 保持在线且不归档；窗口结束再次验证两端健康和提交一致，才执行旧仓库只读归档。
