做出两个能访问的博客并不难，难的是让它们在半年后仍然属于**同一个系统**。

这次搭建个人网站，我最初有两个版本：一个部署在 Vercel，保留 Next.js 的运行时能力；另一个导出为静态文件，部署在 GitHub Pages。它们都能打开，也都有 Blog、Tags、Projects 和 About。仅看页面，我似乎已经拥有了一份主站和一份备份。

但从第一性原理看，这个判断站不住脚。两个网站如果需要分别改文章、项目和身份信息，只是两份会逐渐漂移的副本。真正的可恢复性，不是“另一个 URL 还活着”，而是它能否在需要时提供同一批已经发布的内容。

![单一内容源驱动两套独立应用的架构图](/content/blog/blog-system-architecture.svg)

*图 1：共享的是公开语义内容，不是整套应用实现。*

## 我一开始拥有的，不是灾备

Vercel 版和 Pages 版最初位于两个仓库。它们的视觉、文章和项目数据各自维护，发布链也互不认识。这样的结构适合比较两条技术路线，却不能承诺故障接管。

原因很简单：灾备首先要回答“恢复什么”。我要保护的不是某一份 CSS，也不是某个平台的构建日志，而是已经公开的文章、项目、身份和可追溯的 Git 历史。如果这些内容存在两个可编辑来源，那么每一次更新都会制造新的同步问题。

所以我先接受了一个看似强硬、实际很省事的约束：**所有公开语义内容只有一个来源。** 文章、项目、作者信息和页面正文进入 `packages/content`；两套应用只负责用各自的方式把它们呈现出来。

## 我先写下四个不变量

架构讨论很容易陷入平台功能比较。对个人博客来说，更有效的方法是先写下不能被破坏的结果：

1. 一篇文章只编辑一次，两套网站必须消费同一内容。
2. Vercel 和 Pages 可以分别失败，不能让温备平台反过来阻塞主站发布。
3. 已经成功公开的版本不能在恢复时丢失，也就是已发布版本的 RPO 为 0。
4. 主站持续故障时，人工确认并给出备用入口的 RTO 不超过一小时。

这些不变量把很多“听起来更稳”的方案排除了。例如，让 Pages 先上线、验证后再人工提升 Vercel，确实能得到严格顺序，却会让每一篇普通文章都多一个发布按钮；Pages 一旦故障，主站也无法更新。这对个人博客是过度耦合，不是可靠性。

## 为什么保留两套应用

统一内容不等于统一实现。

Vercel 版可以继续使用 Route Handler、图片优化和平台分析；Pages 版必须接受静态导出、`basePath` 和无运行时服务器的限制。如果为了“代码只有一份”而把这些差异塞进同一个应用，最终会得到大量平台条件分支，也会让静态平台的约束污染主站体验。

我最后采用的是单一 pnpm monorepo：`apps/vercel` 和 `apps/pages` 仍是两套独立 Next.js 应用，`packages/content` 是唯一内容源。两边拥有不同的组件、样式、搜索实现和构建产物，但文章 slug、标签、项目、邮箱和正文必须一致。

这不是追求最高代码复用率，而是在维护效率与故障隔离之间画一条清楚的线。

## “同时发布”不是原子发布

另一个容易混淆的词是“同时”。现在一次合并会同时触发 Vercel 与 Pages，但两个平台不会在同一毫秒完成。

![同一 main 提交触发两条独立部署链](/content/blog/independent-release-flow.svg)

*图 2：发布意图相同，完成时间可以不同，最终由版本端点证明收敛。*

我把 `main` 定义为两版共同的生产意图。合并前，两套应用必须同时通过内容审计、类型检查和生产构建；合并后，它们独立部署。每个站点都暴露 `/version.json`，记录源提交、构建时间和应用类型。

正常情况下，两边应在十五分钟内收敛到同一个提交。部署窗口内短暂不同是状态，不是数据丢失；超过十五分钟仍未追平，才是需要处理的新鲜度故障。

这个表述比“Pages 最多落后一个版本”更诚实。只要不阻塞主站发布，温备平台在故障期间就可能落后多个中间提交。真正重要的是它能否恢复到最新已经发布的内容，以及偏差是否可见。

## 架构图之外，真正花时间的是这些细节

迁移过程中，最费时间的并不是把目录搬进 `apps/`，而是那些很小、却会让“看起来可用”变成“实际不可用”的边界。

Pages 的项目站点不部署在域名根路径。仓库从 `original-blog-vercel` 重命名为 `original-blog` 后，Git 仓库地址可以重定向，项目型 Pages URL 却不会替我自动迁移。`basePath`、静态资源、Feed、Sitemap、搜索索引和本地预览必须一起切换到 `/original-blog`。

移动导航也给了我一次很直接的提醒：桌面端看起来正确，并不代表 320×568 的屏幕能看到全部入口。菜单需要处理滚动锁定、焦点进入、Escape 关闭、导航后复位和桌面宽度恢复，最后再用 Playwright 固化成回归测试。

*图 3：Vercel 主站保留完整的运行时体验和桌面叙事。*

![GitHub Pages 温备站的移动导航](/content/blog/pages-mobile-navigation.png)

*图 4：温备站不只是能返回 200，它在移动设备上也必须可操作。*

Node 版本同样不能靠“应该兼容”带过。仓库、GitHub Actions 和 Vercel 都以 Node 24 为正式基线，所以本地最终验收也显式使用 nvm 中的 Node 24。Node 26 也许能运行部分命令，但它的结果不能代表生产链已经被验证。

还有一个更隐蔽的问题：迁移后残留的 `tsconfig.tsbuildinfo` 曾经让旧工作树假装类型检查通过，也让新工作树出现不一致错误。删除增量缓存后，真正缺失的 React、MDX 类型才暴露出来。缓存可以加速验证，也可以掩盖验证对象已经改变的事实。

## 我如何证明它真的完成了

本地 `build` 成功只是起点。最终证据包括：

- 单一内容源审计确认两套应用没有维护第二份文章和项目；
- Node 24 下格式、类型、双生产构建、静态审计和端到端测试全部通过；
- 仓库改名后的真实 push 能继续触发 CI、Vercel、Pages 和新鲜度检查；
- 两个 `/version.json` 指向同一个提交；
- 首页、深层文章、特殊字符标签、Feed、Sitemap、搜索和 404 都经过公开 URL 验证；
- 390×844、320×568 和 1280×800 的导航行为经过浏览器复检；
- 人工切换演练确认技术核验远小于一小时，同时承认真实 RTO 还要包含发现、判断和通知时间。

这些证据让“上线”不再是一种感觉，而是一组可以重复检查的事实。

## 个人博客不需要银行级容灾，但需要诚实的边界

我没有为这个网站建设自动 DNS 故障切换，也没有把每次发布做成复杂的人工晋级流程。个人博客的影响不足以抵消误判和长期运维成本。

当前 Pages 更准确的名字仍然是温备站：它持续同步内容，拥有可验证的版本与恢复目标，但因为还没有自定义域名，故障时只能人工传播备用 URL，不能透明接管品牌入口。异地 Git 镜像也仍是后续任务。

旧 Pages 仓库同样没有在新站第一次成功时立刻归档。我要求新架构至少经历一次真实公开内容更新，再观察二十四小时。**你正在阅读的这篇文章，就是这次门禁真正等待的内容。**

## 网站从发布真实内容开始

这次重构最有价值的结果，并不是目录终于整齐，而是以后写一篇文章时，我只需要关心文章本身。一次提交会驱动两套不同实现，失败可以被观察，恢复目标有明确含义。

个人网站不会被一次设计完成。它更像一套缓慢生长的系统：先把内容留下，再把重复劳动和模糊承诺逐步删掉。对我来说，这才是数字花园真正开始运转的时刻。
