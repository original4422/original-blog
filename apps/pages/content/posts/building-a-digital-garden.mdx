## 先决定这里为什么存在

这是一个**示例文章**，用于展示中文正文、标题层级、代码块和链接的最终排版。后续可以直接删除，并换成你的真实内容。

博客的第一版不需要回答所有问题。它只需要建立一种足够轻的节奏：记录、发布、回看，再继续修正。把内容存成普通文本，意味着写作不会被某个后台系统锁住；把页面静态生成，则让站点保持简单、快速并且容易迁移。

> 好的个人网站不是简历的放大版，而是一个人如何观察、选择和工作的可见轨迹。

## 内容先于工具

工具应该退到背景里。理想的发布流程可以短到只有三步：

1. 新建一篇 MDX 文件；
2. 写下标题、日期、摘要与标签；
3. 推送到 GitHub，等待自动部署。

```ts
export const note = {
  status: 'unfinished',
  worthPublishing: true,
};
```

不完整并不等于没有价值。很多好文章最早只是一个准确的问题、一段实验记录，或者一次失败后留下的判断。

### 用一个小指标观察节奏

假设一周内写下的有效笔记数为 $n$，真正发布的文章数为 $p$，那么可以用一个很朴素的比率观察写作节奏：

$$
r = \frac{p}{\max(n, 1)}
$$

它不是绩效指标，只用来提醒我们：系统是否让想法更容易从草稿走向公开。

| 状态 | 目标 | 示例动作 |
| --- | --- | --- |
| 草稿 | 保留问题 | 新建 MDX |
| 校对 | 澄清结构 | 补充标题与链接 |
| 发布 | 留下证据 | 推送并检查生产 URL |

## 让结构能够慢慢变化

当前站点把内容分成 Blog、Tags、Projects 和 About。它们不是永远不变的分类，只是第一组足够稳定的入口。未来如果写作重心变化，可以增加 Notes 或 Uses，也可以把 Projects 收进 About；重要的是底层内容仍然清楚可读。

这篇占位文章到这里结束。替换内容时，可以保留 frontmatter 的字段结构，以便搜索、RSS 和 Sitemap 继续自动工作。
